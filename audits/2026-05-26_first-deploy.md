## Mid-execution findings

### Finding A — chapter-route ownership

Scaffold v4.3.0 (#69) auto-injects `/chapters/[...slug]/` when `routes.chapters: true`. ssm carried the consumer-owned route from the pre-v4.3.0 era. v4.3.0 changelog warned "Astro will error on duplicate routes" — investigation against DML (which has the same dual state and ships live) showed Astro's file-system-routes-win precedence silently picks the consumer route. The error warning was overstated; both files coexist gracefully.

Decision Q15 — **layered long-term cleanup**:
- Layer 1 (this commit, ssm): delete `src/pages/chapters/[...slug].astro`. Commit E.
- Layer 2 (follow-up session): same cleanup for DML.
- Layer 3 (book-scaffold-astro v4.6): docs recipe + validator warning + `create-book` template cleanup. Filed at https://github.com/brandon-behring/book-scaffold-astro/issues/76.

### Finding B — pre-existing `ssm-foundations.brandon-behring.dev` Worker binding

Domain already returned HTTP 200 from a Worker before this session. Investigation: a bare-name Worker called `ssm-foundations` was bound to the custom domain — likely from a prior local `wrangler deploy` against the old `wrangler.toml`. The legacy v4.2.0 served all routes except `/chapters/` (404 — v4.3.0 hadn't added the auto-inject yet at that point).

Decision Q16 — **B2: migrate to person-prefixed Worker name**:
- `wrangler.jsonc` uses `name: "brandon-behring-ssm-foundations"`.
- CI deploys to the new Worker; old Worker keeps serving the custom domain (zero downtime).
- User rebinds the custom domain in CF dashboard between Pass 1 and Pass 2 (Phase 4).
- Decision Q17: delete the orphan `ssm-foundations` Worker at session end after Pass 2 verifies (Phase 9.5).

### Finding C — deploy CI failure (validate exit 25)

First Pass 1 push (`50c48aa`) → CI deploy run `26480642760` → step `Run npm run validate` failed with exit code 25 ("25 unknown bibkey errors"). Root cause:

- `book-scaffold validate` resolves `<Cite key=...>` against `src/data/references.json`
- `src/data/references.json` is gitignored, regenerated by `npm run build:bib` from `bibliography.bib`
- Reusable deploy workflow (`brandon-behring/deploy-workflows/.github/workflows/deploy-astro-worker.yml@main`) runs `npm run validate` STANDALONE — no prebuild hook fires
- ssm's `.env` is tracked AND contains `BOOK_PROFILE=academic`, so CI's validate runs strict academic against a missing references.json → 25 errors

**Why DML's CI hadn't failed:** DML's `web/.env` is gitignored at DML's repo root (different `.gitignore` policy from ssm). CI checkout has no `BOOK_PROFILE` set → `book-scaffold validate` defaults to `profile=minimal` → no bibkey enforcement → silent no-op. DML's CI log: `"validate: ✓ 1 chapter(s) checked (profile=minimal); no errors."` — the academic gate has never actually run in CI.

Fix: `ci:validate` npm script wraps validate with required prereqs.

- ssm (commit `dcb1b49`): `"ci:validate": "npm run build:bib && npm run build:labels --if-present && npm run validate"`. `validate-command: validate` → `validate-command: ci:validate` in deploy.yml.
- DML (commit `d23c3b0` on `brandon-behring/double_ml_time_series` — sibling fix for the latent gap, per session decision to patch DML same session): inlines `BOOK_PROFILE=academic` since DML's `.env` is gitignored. After this, DML's CI actually enforces academic-strict for the first time.

Verified locally + in CI for both repos. `ci:validate` is a workaround; the long-term-clean answer (proposed in v4.6 issue) is the `prevalidate` npm hook convention.
